## 二、硬伤与风险（必须修）

### 🔴 硬伤1：日柱计算不准确 — 没用农历/节气

**当前实现**（bazi.ts 第131-142行）：

```typescript
function getDayGanZhi(year: number, month: number, day: number): [string, string] {
  const baseDate = new Date(2000, 0, 1); // 简化为甲子日
  const diffDays = Math.floor((targetDate.getTime() - baseDate.getTime()) / ...);
  const ganIdx = ((diffDays % 10) + 10) % 10;
  const zhiIdx = ((diffDays % 12) + 12) % 12;
  return [TIAN_GAN[ganIdx], DI_ZHI[zhiIdx]];
}
```

**问题**：
- 以2000-01-01为"甲子日"是**错误的**——2000年1月1日实际是甲子日没错，但这只是巧合，基准日应该精确校验
- 更严重的是：**日柱应该以农历/节气为基准，不是公历**。公历直接算日柱在某些边界日期会错一天
- 没有处理闰月、节气交接等边界情况

**影响**：日柱错了 → 日主错了 → 喜用神错了 → **核心卖点崩塌**

**建议**：
- **方案A（推荐）**：集成 `lunar-javascript`（lunar-python的JS版，MIT许可），替换整个排盘引擎
- **方案B**：后端用Python的 `lunar-python`，前端只做展示

### 🔴 硬伤2：年柱没以立春为界

**当前实现**（bazi.ts 第115-119行）：

```typescript
function getYearGanZhi(year: number): [string, string] {
  // 以立春为界简化：直接用年份计算
  const ganIdx = (year - 4) % 10;
  const zhiIdx = (year - 4) % 12;
  return [TIAN_GAN[ganIdx], DI_ZHI[zhiIdx]];
}
```

**问题**：注释写了"以立春为界简化"，但代码**直接用公历年份**，完全没判断立春。1月1日到立春前的出生者，年柱是错的。

**影响**：1-2月出生的人年柱大概率错误。虽然年柱对喜用神影响小于日柱和月柱，但用户如果拿结果去跟其他工具比对，会直接发现不一致，损失信任。

**建议**：必须加立春判断。lunar-javascript自带节气计算。

### 🔴 硬伤3：月柱用公历月份近似农历

**当前实现**（bazi.ts 第121-130行）：

```typescript

  const zhiIdx = (month + 1) % 12; // 简化映射
  // ...

```

**问题**：月柱必须以**节气**为界（立春→寅月，惊蛰→卯月……），不是以公历月份近似。1月≠寅月。

**影响**：月柱对喜用神判断影响极大（月令是格局判定的核心），月柱错了 → 格局错了 → 喜用神大概率也错。

### 🟡 风险4：喜用神推导过于简化

**当前实现**（bazi.ts 第71-84行）：


**问题**：

- 没有"从格"判断（从旺格、从弱格的喜用神与普通格局完全相反）

**影响**：喜用神推导是核心差异化卖点。90%竞品只会"缺啥补啥"，但如果我们的"喜用神分析"也只是简化版的扶抑法，差异化就没那么大了。
