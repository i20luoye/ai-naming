## 一、总体评价

**评分：7.5/10**

这是一份**完成度相当高**的 MVP 代码。视觉设计、前端交互、品牌一致性做得很出色，Dark Mode 沉浸感、金墨配色、动画节奏都可圈可点。但核心技术层（八字排盘、AI 起名/测名）存在若干准确性和架构层面的问题，直接上线有风险。

**一句话总结：** 皮相惊艳，骨架扎实，血肉需补——前端 85 分，后端/算法 60 分。

## 三、用户体验与流程设计 ⚠️

### 3.1 做得好

- **4 步引导式流程**（输入→排盘→起名→结果）：步骤指示器贯穿全流程，降低认知负担。
- **AI 前奏动画**（`name/result/page.tsx`）：4 步推演过程可视化，"跳过"按钮让不耐烦的用户快速到达结果，体验设计有巧思。
- **信息回显条**：结果页顶部回显姓氏/性别/日期/日主/喜用神，用户不需要来回翻看。
- **对比摘要条**：5 个名字的横向对比栏，点击可滚动到对应卡片，交互贴心。
- **付费墙设计**：模糊遮罩+渐变蒙层+解锁按钮，不突兀但存在感足够。

### 3.2 可改进

- **输入页 4 步过长**（`name/input/page.tsx`）：姓氏→性别→生辰→方位，4 个子步骤在一个页面内切换。实际产品中，用户对"方位"（出生地省/市）的需求感知弱，建议合并为 2-3 步，方位设为可选项。
- **排盘页信息密度过高**（`name/bazi/page.tsx`）：四柱八字 + 五行雷达图 + 纳音 + 喜用神 + 格局——对普通用户来说信息过载。建议折叠次要信息（如纳音、藏干详情），默认只展示四柱+五行+喜用神。
- **结果页无"重新生成"按钮**：用户只能"修改偏好"或"重新起名"（回到第一步），没有"基于当前结果重新生成另一批名字"的快捷入口。
- **缺少 loading 骨架屏**：排盘和起名 API 调用期间，页面无明显 loading 状态（除 AI 前奏动画外），网络慢时用户可能以为卡死。

## 四、八字算法准确性与完整性问题 🔴

### 4.1 关键问题（必须修）

#### 问题 1：年柱计算未处理立春分界

```typescript
// src/lib/bazi.ts 第 120 行
const ganIdx = (year - 4) % 10;
const zhiIdx = (year - 4) % 12;
```

这段代码**直接用公历年份计算年柱**，完全没有考虑立春。八字中年柱以立春为界，如 2025 年 2 月 3 日（立春前）出生仍属甲辰年而非乙巳年。**这是八字排盘的根本性错误**，会导致年柱、月柱全部错误。

**修复建议：** 需要实现公历→农历转换（含节气判断），或集成成熟的开源库（如 `lunar-javascript`）。

#### 问题 2：月柱计算严重简化

```typescript
// src/lib/bazi.ts 第 131 行
const zhiIdx = (month + 1) % 12; // 简化映射
```

月支固定为寅月=2 月、卯月=3 月……这同样是**没有考虑节气分界**。正月（寅月）从立春到惊蛰，而非公历 2 月整月。且月干使用年上起月法时的 base 计算逻辑也有问题。

#### 问题 3：日柱计算基准日错误

```typescript
// src/lib/bazi.ts 第 143 行
const baseDate = new Date(2000, 0, 1); // 2000年1月1日 = 庚辰年丙子月甲子日 (简化)
```

注释写了"简化为甲子日"，但 2000 年 1 月 1 日实际上是**癸亥日**，不是甲子日。差一天会导致所有日柱偏移 1 天。

#### 问题 4：时柱计算的天数偏移

`getHourGanZhi` 接收 `dayGan` 参数，但日上起时法需要的是当天日柱的天干，目前传入的是正确的 `dayGan`，这一点是对的。但时支索引计算：
```typescript
const zhiIdx = Math.floor(((hour + 1) % 24) / 2);
```
对于 `hour=23`，`(23+1)%24=0`，`0/2=0` → 子时，正确。对于 `hour=0`，`(0+1)%24=1`，`1/2=0` → 子时，也正确。这个实现是对的。

#### 问题 5：格局判定过于简化

```typescript
// src/lib/bazi.ts determinePattern()
if (WUXING_SHENG[dayWx] === monthWx) return '食神格';
```

仅根据月令天干与日主的关系判定格局，完全忽略了地支藏干中透出的十神，以及月令本气/中气/余气的取格规则。真实的子平格局判定远比这复杂。

#### 问题 6：喜用神推断过于简化

```typescript
if (strength === '弱') {
  return { xiYong: [WUXING_BEI_SHENG[dayWx], dayWx], ... };
}
```

仅凭五行数量比例（>=45%为旺）判断强弱，然后给出喜用神。真实的八字喜用神判定需考虑：调候、通关、格局成败、干支组合等，远非简单的旺弱二分。

### 4.2 影响评估

| 问题 | 严重程度 | 影响范围 | 用户感知 |
|------|---------|---------|---------|
| 年柱未考虑立春 | 🔴 严重 | 1-2月出生者年柱月柱全错 | 用户不知情但结果错误 |
| 月柱未考虑节气 | 🔴 严重 | 所有用户月柱可能偏移 | 同上 |
| 日柱基准日错误 | 🟡 中等 | 所有日柱偏移1天 | 同上 |
| 格局/喜用神简化 | 🟡 中等 | 起名方向可能不准 | 专业用户可能质疑 |
| 缺少真太阳时 | 🟢 低 | 偏远地区时辰偏差 | 几乎无感知 |

### 4.3 修复方案建议

**短期（上线前）：**
1. 集成 `lunar-javascript` 或 `lunar-typescript` 开源库处理公历→农历+节气转换
2. 修正日柱基准日
3. 在排盘页底部加醒目的免责声明

**中期（上线后迭代）：**
4. 引入真太阳时校正（根据出生地经度调整时辰）
5. 完善格局判定（至少支持正格八格）
6. 喜用神增加调候维度

## 五、AI 起名/测名质量 ⚠️

### 5.1 做得好

- **System Prompt 设计专业**：`generate-names/route.ts` 的提示词包含命名规则、输出格式、合规约束，覆盖全面。
- **测名维度丰富**：`test-name/route.ts` 覆盖了喜用神匹配、三才五格、音韵、谐音、重名、寓意、逐字解析等 9 个维度。
- **JSON 输出 + 正则提取容错**：`jsonMatch` 正则兜底，避免 LLM 格式偏差导致解析失败。

### 5.2 关键问题

#### 问题 1：前端名字池是 Mock 数据，不走 AI API

这是最核心的问题。`name/result/page.tsx` 中的 `generateVisibleAndLocked()` 函数使用的是硬编码的 `NAME_POOL`（20 个固定名字），通过喜用神匹配度排序。**AI API（`/api/generate-names`）只用于生成分析文字，实际名字来自前端 Mock 数据池。**

这意味着：
- 无论用户什么八字，看到的都是同一批名字（只是排序不同）
- "AI 起名"变成了"从前端池子里挑名字"
- 名字不会随八字变化而变化

这是一个**产品定位级别的偏差**——用户以为 AI 在起名，实际是静态匹配。

#### 问题 2：测名 API 无缓存/限流

`test-name/route.ts` 每次调用都请求 LLM，无任何缓存或限流机制。测名是高频场景（用户可能反复测试多个名字），LLM 调用成本会快速增长。

#### 问题 3：LLM 返回的诗词出处可能造假

System Prompt 要求"诗词出处必须真实可查"，但 LLM 自身无法验证真实性。`doubao-seed-2-0-lite-260215` 模型可能编造诗词出处。建议增加后处理校验或使用诗词数据库交叉验证。

### 5.3 修复建议

| 优先级 | 问题 | 建议 |
|--------|------|------|
| P0 | 名字来自 Mock 池 | 改为调用 AI API 实时生成名字，前端名字池作为 fallback |
| P1 | 测名无缓存 | 对同一名字+八字组合缓存结果（Redis/内存），避免重复调用 |
| P1 | 诗词出处真实性 | 增加诗词出处校验模块（可离线跑一次验证） |
| P2 | 起名 API 无超时控制 | LLM 调用增加 timeout（当前无），防止用户无限等待 |

### 8.2 风险点

1. **合规风险**：虽然 System Prompt 有合规约束，但 LLM 输出不可控，可能偶尔违反合规要求。建议增加输出侧关键词过滤。
2. **LLM 成本风险**：无鉴权 + 无限流 + 无缓存 → 恶意调用可导致巨额 API 费用。
3. **数据丢失风险**：所有数据存储在 localStorage，用户换设备/清缓存即丢失。
4. **八字准确性风险**：如果用户发现八字排盘不准（因节气/立春问题），会在社交媒体传播负面评价，品牌声誉受损。
