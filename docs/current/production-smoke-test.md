## 2. 线上环境变量清单

### 2.1 必须配置的环境变量

| 变量名 | 说明 | 示例值（非真实） | 必填 |
|---|---|---|---|
| `NEXT_PUBLIC_SITE_URL` | 生产站点 URL，用于 sitemap/canonical/JSON-LD | `https://tianyan.app` | ✅ 必填 |
| `NEXT_PUBLIC_APP_ENV` | 应用环境标识 | `production` | ✅ 必填 |
| `LLM_API_KEY` | LLM 服务 API 密钥 | `sk-xxxxxxxx`（不写真实 key） | ✅ 必填 |
| `LLM_BASE_URL` | LLM API 基础 URL | `https://api.example.com/v1` | ✅ 必填 |
| `LLM_MODEL` | LLM 模型名称 | `agnes-2.0-flash` | ✅ 必填 |

### 2.2 安全要求

- **不写真实 key**：本文档、代码注释、commit message 中均不得出现真实 API key
- **`.env.local` 不得提交**：`.gitignore` 必须包含 `.env.local`，不得推送到仓库
- **线上平台环境变量必须与本地一致**：变量名、格式必须完全匹配，值由部署平台环境变量配置注入
- **`NEXT_PUBLIC_SITE_URL` 必须是真实生产域名**：不得是 `localhost`、`127.0.0.1` 或内网 IP
- **客户端不暴露 secret**：`LLM_API_KEY` 不得出现在客户端 bundle 中

### 2.3 环境变量验证方法

```bash
# 检查 NEXT_PUBLIC_SITE_URL 是否为生产域名
curl -s https://${DOMAIN}/sitemap.xml | head -5

# 检查客户端 bundle 不含 LLM_API_KEY
# 在浏览器 DevTools → Sources → Search 搜索 "LLM_API_KEY" 或 "sk-"
# 搜索结果应为空

# 检查 .env.local 不在 git 中
git ls-files | grep -i "\.env"
# 不应出现 .env.local
```

## 3. 线上 Smoke Test URL 清单

部署完成后，对以下每个 URL 执行 smoke test：

| URL | 期望状态 | 检查项 |
|---|---|---|
| `/` | HTTP 200 | 首页不白屏、title 含"天衍"、description 正常、移动端可读、不暴露 secret、无合规风险词 |
| `/name/input` | HTTP 200 | 起名输入页不白屏、表单可交互、移动端可用 |
| `/test-name` | HTTP 200 | 测名页不白屏、表单可交互、移动端可用 |
| `/privacy` | HTTP 200 | 隐私政策不白屏、含"传统文化参考"、含"不提供"、无正向承诺 |
| `/terms` | HTTP 200 | 服务条款不白屏、含"传统文化参考"、含"不提供" |
| `/disclaimer` | HTTP 200 | 免责声明不白屏、含"传统文化参考"、含"不构成" |
| `/sitemap.xml` | HTTP 200 | 返回 XML、含首页/输入/测名/合规页、不含 /api/、不含结果页 |
| `/robots.txt` | HTTP 200 | 返回 text、含 `Disallow: /api/`、含 sitemap 指向 |

### 每个 URL 的通用检查项

- [ ] HTTP 200（非 404/500/502）
- [ ] 页面不白屏（有可见内容）
- [ ] `<title>` 标签正常（含"天衍"）
- [ ] `<meta name="description">` 正常
- [ ] 移动端可读（viewport meta 存在）
- [ ] 不暴露 secret（页面源码无 API key）
- [ ] 不出现明显合规风险词（"算命""改运""保证旺财"作为服务描述）

## 5. 线上 LLM 检查

### 5.1 凭证配置检查

| 场景 | 操作 | 预期结果 |
|---|---|---|
| 有 LLM_API_KEY | 正常调用 `/api/generate-names` | 返回 200 + 名字数组 |
| 缺 LLM_API_KEY | 移除环境变量后调用 | 返回 503 + `LLM_CREDENTIALS_MISSING` + "AI 服务未配置，请稍后再试" |
| 缺 LLM_API_KEY（测名） | 移除环境变量后调用 `/api/test-name` | 返回 503 + `LLM_CREDENTIALS_MISSING` |

### 5.2 错误兜底检查

| 场景 | 预期结果 |
|---|---|
| LLM 上游 500 | 返回 500 + "名字生成失败，请稍后重试"，**不暴露 stack trace** |
| LLM 上游 500（测名） | 返回 500 + "测名评分失败，请稍后重试" |
| LLM 返回格式异常 | 返回 502 + "AI 返回格式解析失败"，**不暴露 raw AI output** |
| LLM 返回格式异常（测名） | 返回 502 + "测名结果解析失败，请稍后重试"，**不返回 raw** |

### 5.3 安全保证

- **不暴露 API key**：所有错误响应体不含 `LLM_API_KEY` 值
- **不暴露 raw prompt**：所有错误响应体不含发送给 LLM 的 prompt 原文
- **不暴露 raw AI output**：502 响应不含 `raw` 字段
- **不暴露 stack trace**：500 响应只返回中性文案

### 5.4 上线后质量验证

上线后至少手动生成 3 组不同案例，确认质量正常：

| 案例 | 姓氏 | 性别 | 出生信息 | 检查项 |
|---|---|---|---|---|
| 案例 1 | 林 | 男 | 2024-03-15 14:00 | 名字正常生成、有五行分析、无合规风险词 |
| 案例 2 | 王 | 女 | 2023-11-20 08:00 | 名字正常生成、有出处说明、质量分正常 |
| 案例 3 | 张 | 男 | 2024-06-01 22:00 | 名字正常生成、基础候选/知识库校验标签正确 |

## 6. 3 天小流量测试计划

### Day 1：自测（10 次完整流程）

| 时段 | 任务 | 记录项 |
|---|---|---|
| 上午 | 完整跑 5 次起名主流程 | 每次是否成功、是否有异常、生成耗时 |
| 下午 | 完整跑 5 次测名流程 | 每次是否成功、是否有异常、生成耗时 |
| 晚上 | 汇总失败点 | 列出所有失败点，标注严重程度（阻断/非阻断） |

**Day 1 通过标准**：无阻断问题（白屏、500、数据丢失、合规风险词）。

### Day 2：熟人测试（5-10 人）

| 任务 | 收集项 |
|---|---|
| 邀请 5-10 个熟人测试 | 每人完成 1 次起名 + 1 次测名 |
| 观察用户行为 | 是否能独立完成流程、是否在某个步骤卡住 |
| 收集反馈 | 是否愿意收藏名字、是否愿意点击「完整报告」、对价格的接受度 |
| 记录异常 | 任何报错、白屏、体验问题 |

**Day 2 通过标准**：≥80% 测试者能独立完成主流程，无阻断问题。

### Day 3：修复与决策

| 任务 | 产出 |
|---|---|
| 根据反馈修复阻断问题 | 修复清单 + 验证结果 |
| 评估非阻断问题 | 是否需要立即修复、可延后修复 |
| 决策是否进入下一阶段 | □ 进入真实支付设计阶段 □ 暂缓，继续优化 |

**Day 3 决策标准**：
- 无阻断问题 → 可进入真实支付设计阶段
- 有阻断问题但已修复 → 修复后重新跑 1 天自测，无新阻断则进入
- 有未修复阻断问题 → 暂缓，继续优化

## 7. 回滚预案

| 场景 | 触发条件 | 操作 |
|---|---|---|
| LLM API 大面积失败 | 连续 3 次请求失败率 >50% | 保持服务运行，fallback 到基础推荐池；检查 LLM 服务状态 |
| 前端页面白屏 | 首页或核心页面白屏 | 检查构建产物，回退到上一版本部署 |
| 合规风险 | 发现合规风险词出现在线上 | 立即下线相关页面，审查内容后重新上线 |
| 流量异常 | 单 IP 请求量超限 | 检查限流配置（当前 10 次/60 秒/IP），必要时临时降低限流阈值 |
| 数据泄露 | 发现 API key 或敏感信息泄露 | 立即轮换密钥，审查日志，修复泄露点后重新部署 |

### 回滚操作步骤

1. 在部署平台找到上一版本部署记录
2. 触发回滚（Redeploy / Rollback）
3. 验证回滚后服务正常
4. 记录事故原因和修复方案
