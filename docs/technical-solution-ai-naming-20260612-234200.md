# AI起名工具 技术方案

> 状态: Draft
> 日期: 2026-06-12
> 负责人: 全栈研发Agent
> 置信度: medium（部分容量数据待验证）

## 1. 方案摘要

- **推荐架构**: Next.js 14 (App Router) 全栈应用 + 微信小程序双端，Supabase (PostgreSQL) 数据层，Vercel 部署 Web 端
- **关键决策**:
  - 排盘引擎选 lunar-javascript (MIT, 前端可跑) + bazi-calculator-by-alvamind (TypeScript, 分析层补充)
  - 喜用神核心算法自建，参考 china-testing/bazi (Python→TypeScript 移植) + korean-saju 逻辑
  - AI层用 DeepSeek V3 API (性价比) + OpenAI GPT-4o-mini (备选)，SSE 流式输出
  - 产品定位"传统文化数字化工具"，合规模块内建于架构层
- **预期收益**: MVP 1人2周可交付，核心差异化（喜用神精准分析）技术可行
- **最大风险**: 喜用神算法准确率需命理顾问验证；合规审查可能要求产品措辞调整
- **评审阻塞项**: 喜用神算法验证结果未出前，格局分析模块不可进入开发

## 2. 背景与目标

| 项 | 内容 |
| --- | --- |
| 业务背景 | 八字起名细分市场 52.7 亿元（2025），付费转化率 8-12%，客单价 35→128 元升级路径清晰。90% 竞品做"缺啥补啥"的错误五行补益，市场存在专业度空位 |

| 成功标准 | MVP 阶段：(1) 八字排盘准确率≥99%（与 lunar-javascript 交叉验证）；(2) 喜用神判定与命理师一致率≥85%；(3) AI 起名响应时间≤5s（流式首字≤1s）；(4) 合规词过滤覆盖率100%；(5) 付费转化率≥8% |

## 6. 方案选型与决策

### 6.1 关键选型矩阵

#### 决策1：排盘引擎

| 评估标准 | 权重 | lunar-javascript | lunisolar | bazi-calculator |
| --- | ---: | ---: | ---: | ---: |
| 功能完整度 | 25 | 5 | 4 | 4 |
| TypeScript 支持 | 20 | 3 | 5 | 5 |
| 社区成熟度 | 20 | 5 | 2 | 3 |
| 前端可运行 | 15 | 5 | 5 | 5 |
| 协议友好度 | 10 | 5 | - | 5 |
| 维护活跃度 | 10 | 4 | 3 | 3 |

**推荐**: `lunar-javascript`（主排盘）+ `bazi-calculator-by-alvamind`（分析层补充）
**理由**: lunar-javascript 功能最全（5k+ stars，MIT，6tail 系列多语言），bazi-calculator 提供更高层的 TypeScript 分析能力，两者互补
**放弃方案**: lunisolar（社区太小，长期维护风险）
**反转条件**: 如果 TypeScript 原生支持成为硬性要求且 lunisolar 社区成长

#### 决策2：喜用神算法实现路线

| 评估标准 | 权重 | 自建(参考china-testing/bazi) | 移植korean-saju | 调用bazi-pan-skill |
| --- | ---: | ---: | ---: | ---: |
| 算法准确度潜力 | 30 | 5 | 4 | 3 |
| TypeScript 兼容 | 25 | 4 | 2 | 2 |
| 可控性 | 20 | 5 | 3 | 2 |
| 实现速度 | 15 | 2 | 3 | 4 |
| 验证基础 | 10 | 3 | 5 | 2 |

**推荐**: 自建喜用神引擎，核心逻辑参考 china-testing/bazi（Python→TypeScript 移植），korean-saju 的验证方法论作为质量参考
**理由**: 自建完全可控，TypeScript 原生，可针对产品需求定制；china-testing/bazi 格局分析最全面；korean-saju 的 35000 例验证数据可做回归测试
**放弃方案**: 直接调用 Python 服务（增加运维复杂度）、调用 MCP/API（依赖外部服务）
**反转条件**: 如果移植工作量远超预期（>10 人天），考虑先上 MCP 调用方案快速验证

#### 决策3：AI 模型选择

| 评估标准 | 权重 | DeepSeek V3 | GPT-4o-mini | 通义千问 |
| --- | ---: | ---: | ---: | ---: |
| 中文创作质量 | 30 | 4 | 4 | 4 |
| API 成本 | 25 | 5 | 3 | 4 |
| 响应速度 | 20 | 4 | 5 | 4 |
| 流式支持 | 15 | 5 | 5 | 5 |
| 稳定性 | 10 | 3 | 5 | 4 |

**推荐**: DeepSeek V3（主力）+ GPT-4o-mini（备选），API 层做统一抽象
**理由**: DeepSeek 中文能力强+成本极低，但稳定性略逊；GPT-4o-mini 作为 fallback
**反转条件**: 如果 DeepSeek API 稳定性不达标（可用性<99%），切换为通义千问

#### 决策4：部署架构

| 评估标准 | 权重 | Vercel + Supabase | 自建服务器 | 云函数 |
| --- | ---: | ---: | ---: | ---: |
| 开发效率 | 25 | 5 | 2 | 4 |
| 运维成本 | 25 | 5 | 2 | 4 |
| 弹性扩展 | 20 | 4 | 3 | 5 |
| 数据主权 | 15 | 3 | 5 | 3 |
| 合规可控 | 15 | 3 | 5 | 3 |

**推荐**: Vercel + Supabase（MVP 阶段），后续数据合规需求增强时迁移
**理由**: 独立开发者1人团队，Vercel+Supabase 零运维；用户生辰数据72h自动删除策略降低合规风险
**放弃方案**: 自建服务器（运维成本高）、纯云函数（调试复杂）
**反转条件**: 如果合规审查要求所有数据必须存储在境内服务器

### 6.2 决策汇总

| 决策点 | 推荐方案 | 放弃方案 | 选择理由 | 反转条件 | 置信度 |
| --- | --- | --- | --- | --- | --- |
| 排盘引擎 | lunar-javascript + bazi-calculator | lunisolar 单独用 | 功能最全+社区最成熟 | TS原生硬性要求 | high |
| 喜用神算法 | 自建(参考china-testing/bazi) | Python服务调用 | 可控+TypeScript原生 | 移植>10人天 | medium |
| AI模型 | DeepSeek V3 + GPT-4o-mini备选 | 单一模型 | 成本+质量平衡 | DeepSeek稳定性<99% | medium |
| 部署 | Vercel + Supabase | 自建服务器 | 零运维+快速迭代 | 数据出境合规要求 | medium |
| 前端框架 | Next.js 14 App Router | Nuxt / Remix | 生态+SSR+API Routes | - | high |

### 7.3 Runtime Flow（核心运行流程）

#### 流程1：AI起名主流程

```
用户输入生辰 → [1]格式校验 → [2]公历转农历(lunar-js) → [3]八字排盘(lunar-js+bazi-calc)
     → [4]五行分布+日主强弱分析 → [5]喜用神判定(自建引擎)
     → [6]喜用神+字库筛选候选字(五行属性匹配+笔画数+吉凶过滤)
     → [7]AI生成名字(DeepSeek, 基于候选字+诗词出处+寓意方向)
     → [8]合规过滤(替换违规词+注入免责声明)
     → [9]名字质量评估(五行补益度+三才五格+谐音+热名)
     → [10]SSE流式返回结果
```

#### 流程2：喜用神判定子流程

```
四柱八字输入 → 日主识别 → 格局判断(正官/偏官/正印/偏印/食神/伤官/正财/偏财/比肩/劫财)
     → 日主强弱分析(得令/得地/得生/得助 四维度评分)
     → 用神选取策略:
        - 身强: 克泄耗 (官杀/食伤/财星)
        - 身弱: 生助 (印星/比劫)
        - 调候: 寒暖燥湿平衡
        - 通关: 两行交战取中间
     → 喜用神输出 (喜神/用神/忌神/仇神/闲神)
```

#### 流程3：支付解锁流程

```
免费报告(基础排盘+2个名字) → 用户点击"解锁完整报告"
     → 微信支付/Stripe → 支付回调 → 解锁完整报告(10个名字+AI详细解读+名字评估)
     → 72h 后自动删除生辰数据
```

## 8. 关键模块设计

### 8.1 喜用神引擎（核心差异化模块）

| 项 | 内容 |
| --- | --- |
| **职责** | 接收八字四柱，输出喜用神判定结果（喜神/用神/忌神/仇神/闲神） |
| **非职责** | 不负责排盘计算、不负责名字生成、不直接调用LLM |
| **接口** | `XiYongEngine.analyze(bazi: BaziResult): XiYongResult` |
| **数据** | 格局规则表（10种正格局+特殊格局）、调候用神表（穷通宝鉴）、通关规则 |
| **核心逻辑** | (1)日主强弱评分(得令30%+得地25%+得生25%+得助20%) → (2)格局判断(十神组合匹配) → (3)用神选取(身强克泄耗/身弱生助/调候/通关) → (4)喜忌神推导 |
| **可观测性** | 分析耗时、格局分布统计、与基础模式(缺啥补啥)结果偏差率 |
| **安全** | 输入校验(八字合法性)、输出注入"仅供参考"标签 |
| **测试** | 50+典型八字命理顾问验证 + 35000例korean-saju回归测试 + 与"缺啥补啥"对比测试 |

### 8.2 AI 起名服务

| 项 | 内容 |
| --- | --- |
| **职责** | 接收喜用神结果+用户偏好，通过LLM生成5-10个候选名字，SSE流式输出 |
| **非职责** | 不负责排盘、不负责喜用神计算、不负责支付 |
| **接口** | `POST /api/naming/generate` (SSE) |
| **核心逻辑** | (1)基于喜用神筛选候选字库(五行属性+笔画+吉凶) → (2)构建Prompt(喜用神+候选字+用户偏好+诗词出处) → (3)调用DeepSeek V3流式生成 → (4)合规过滤 → (5)名字质量评估打分 |
| **可观测性** | 生成耗时(P50/P95/P99)、模型切换次数、合规过滤命中率 |
| **安全** | Prompt注入防护、输出内容二次合规扫描、速率限制(每用户每天3次免费) |
| **测试** | 名字创意质量人工评审 + 合规词100%拦截测试 + 流式输出稳定性测试 |

### 8.3 合规过滤层

| 项 | 内容 |
| --- | --- |
| **职责** | 所有面向用户的输出必须经过合规过滤，自动替换违规词+注入免责声明 |
| **非职责** | 不负责业务逻辑 |
| **接口** | `ComplianceFilter.scan(text: string, context: OutputContext): FilteredResult` |
| **核心逻辑** | (1)违规词库匹配替换("算命"→"传统文化分析"，"改运"→"参考建议") → (2)免责声明自动注入 → (3)不确定内容标记"仅供参考" |
| **数据** | compliance_words 表（违规词→替换词映射，分类：算命类/预测类/改运类/玄学类） |
| **可观测性** | 过滤触发率、各分类命中分布、误判率(用户反馈) |
| **安全** | 宁可过度过滤不可遗漏；过滤失败时阻断输出 |
| **测试** | 全量违规词100%拦截 + 500条正常内容零误判 + 端到端合规扫描 |

### 8.4 名字质量评估

| 项 | 内容 |
| --- | --- |
| **职责** | 对生成的名字进行多维度质量评分 |
| **非职责** | 不负责名字生成 |
| **接口** | `NameAssessor.assess(name: string, bazi: BaziResult, xiyong: XiYongResult): AssessmentResult` |
| **核心逻辑** | (1)五行补益度(名字五行vs喜用神匹配度，权重35%) → (2)三才五格评分(权重25%) → (3)谐音检测(权重20%) → (4)热名/网红名检测(权重15%) → (5)综合评分(权重5%调整) |
| **数据** | 热名黑名单(hot_name_blacklist)、谐音库(pinyin-pro) |
| **可观测性** | 评分分布、热名命中率、谐音问题率 |
| **测试** | 100个名字与人工评分一致性≥80% |
