## 二、所有Agent共识（多人指出，标注具体Agent）

### 共识1：八字排盘精度是致命硬伤 — 必须P0修复
**指出者：产品经理Agent、竞品调研Agent、Claude Code、调研分析师、产品运营Agent、全栈研发Agent（6/9个完成评审的Agent）**

具体问题：
- **月柱用公历而非节气**：`bazi.ts` 的 `getMonthGanZhi` 直接用公历月份映射月支，八字必须以节气为界
- **日柱基准日错误**：用 `new Date(2000,0,1)` 作基准日+天数差推算。全栈研发Agent明确指出：**2000年1月1日实际是甲戌日，不是代码中假设的甲子日**，差一天则所有日柱全错
- **年柱没判立春**：1-2月出生年柱应该是前一年的，代码未处理
- **缺真太阳时校正**：不同经度出生的人时辰应调整，API也声称支持但代码完全没实现

**后果共识**：八字算错 → 喜用神算错 → 起名五行方向全错 → 用户信任一次性崩塌

**共识方案**：集成 `lunar-javascript` / `lunar-typescript`（MIT协议），替换整个排盘引擎

| Agent | 表述 |
|-------|------|
| 产品经理Agent | "八字排盘是用户信任的基础，错一个就会导致喜用神完全颠倒" |
| 竞品调研Agent | "日主错→喜用神错→核心卖点崩"，评分7.8/10 |
| Claude Code | "一个'八字起名'产品排盘算错是致命的——用户随便一验就能发现不对" |
| 调研分析师 | "八字计算存在精度问题——影响专业可信度" |
| 产品运营Agent | "最大的问题是八字排盘算法有根本性错误" |
| 全栈研发Agent | "2000年1月1日实际是甲戌日，不是甲子日。基准日错了，所有日柱结果全错" |

## 三、各Agent独有发现（仅该Agent指出）

### 产品经理Agent独有

1. **缺README**：当前README.md是Next.js模板默认内容，完全没有"天衍"产品说明
2. **API层缺数据库持久化**：三个API纯计算+LLM调用，结果不存库。package.json装了 `drizzle-orm` + `@supabase/supabase-js` + `pg`，但完全没用到
3. **缺环境变量管理**：`.env` / `.env.example` 不在仓库中，Supabase/Coze SDK/AWS S3配置方式不明确
4. **缺测试**：package.json没有测试脚本、没有jest/vitest依赖
5. **AGENTS.md可更结构化**：建议增加"常见修改场景"指引
6. **API设计清晰**：入参/出参规范、错误处理、LLM降级策略（JSON解析失败时返回raw）都有考虑 — 正面肯定

### 竞品调研Agent独有

1. **9维度测名是核心亮点**：逐字解析+五行补益路径+音韵分析+方言谐音+重名风险+优化建议，深度远超竞品
2. **P0修复工时评估1-2天**：集成lunar-javascript替换排盘引擎
3. **P1建议**：喜用神补调候+通关、LLM五行后验证、五行字库
4. **P2建议**：魔鬼辩护、名字对比、单元测试、缓存策略

### 增长获客 Agent独有

1. **SEO零基础**：无页面metadata、无sitemap、URL无语义，搜索引擎根本找不到。这是P0中的P0
2. **首页CTA模糊**：用户要走4步才能看到第一个名字，3步成本0步回报。Hero区被大量自嗨内容稀释
3. **无付费墙、无裂变、无埋点**：结果全免费，没有分享机制，没有任何数据衡量漏斗转化
4. **4件最该先做的事（~10h工时）**：
   - 每个页面加SEO metadata + JSON-LD Schema
   - URL改为中文拼音关键词
   - 加sitemap + robots
   - 埋点：页面PV + 流程每步转化率

### 前端设计Agent独有

1. **内嵌全局样式与shadcn/ui冲突**：`.card` 等选择器会与shadcn/ui组件冲突，必须拆组件+样式迁入globals.css
2. **localStorage无错误处理/无版本管理**：隐身模式直接白屏，数据结构变更后旧数据崩溃
3. **package.json name是"projects"**：应该是"ai-naming"或"tianyan"
4. **硬编码色彩**：部分组件直接写色值而非用CSS变量
5. **圆角不一致**：未统一使用DESIGN.md中的圆角规范
6. **images配置过宽**：Next.js图片域名配置过于宽泛
7. **修复工时预估约12.5h**，建议按P0→P1→P2逐步推进

### Claude Code独有

1. **步骤指示器不一致**：显示4步但实际有5个页面，对不上
2. **LLM调用5-10秒无骨架屏/loading**：用户以为卡死了
3. **全是 `'use client'`**：首屏性能可以更好，应区分Server/Client组件
4. **排盘页45KB、结果页50KB**：核心页面文件普遍过大

### 自媒体运营达人独有

1. **定价不一致**：增长策略文档¥9.9 vs PRD¥29.9，必须在MVP前对齐（建议先用¥9.9做A/B测试低价漏斗）
2. **品牌名不一致**："Ficita"在两份文档里出现，PRD合并版没提品牌名，主页和README必须统一
3. **海外华人场景是"PPT层面强"**：但MVP只做了微信小程序，海外用户根本用不了
4. **缺"证据型内容"**：增长策略要发小红书"案例展示"，但没有真实种子用户的案例
5. **真太阳时校正是传播素材**：建议做"对比演示图"（同一时辰北京vs纽约的八字差异）
6. **改名场景V1.2才有**：但改名是真正高客单价场景，建议提前调研
7. **起名数据资产化**：建议每次生成都匿名沉淀，半年后变成竞品难以复制的数据护城河
8. **GitHub仓库呈现建议**：缺LICENSE文件、缺产品截图/GIF、缺中英文README、缺demo链接

### 调研分析师独有

1. **付费解锁区是假的**：结果页"解锁完整方案 ¥29.9"按钮没有支付逻辑，"已有3,862人解锁"是硬编码假数据，可能构成虚假宣传
2. **大量未使用的依赖**（影响包体积~4-5MB）：
   - `@aws-sdk/client-s3`（~2MB）
   - `@supabase/supabase-js`（~500KB）
   - `drizzle-orm`/`drizzle-kit`/`drizzle-zod`（~1MB）
   - `pg`（~300KB）
   - `recharts`（~800KB）
   - `react-day-picker`（~200KB）
3. **无障碍不足**：自定义组件缺少ARIA标签、金色文字在深色背景上对比度需验证（WCAG AA）、缺跳过导航链接
4. **shadcn/ui全量安装但只用了不到1/3**：建议按需清理未使用的组件文件
5. **评分7.5/10**："视觉和体验远超MVP水准，但'里子'（数据真实性、计算精度、代码质量）需要补课"

### 产品运营Agent独有

1. **名字生成未走AI API**：应接入 `/api/generate-names` 的真实调用，NAME_POOL仅作降级方案
2. **LLM输出合规后过滤**：只在system prompt写了合规约束，没有对LLM返回结果做后处理过滤。LLM可能输出"改运""命理"等违规词汇直接传给用户
3. **P1建议**：接入微信支付、用户系统（保存起名记录）、真太阳时校正、测名结果缓存
4. **评分7.5/10**

### 全栈研发Agent独有

1. **日柱基准日精确纠错**：2000-01-01实际是**甲戌日**而非代码假设的甲子日，这是最致命的bug
2. **LLM无合规后过滤**：只在system prompt约束，没有对返回结果做黑名单过滤。PRD明确要求"合规中间件"
3. **72小时自动删除没实现**：首页写了"您的信息仅用于分析，72小时后自动删除"，但代码没有实现任何自动清理机制
4. **localStorage明文存储生辰信息**：不符合PRD的"加密存储"要求
5. **付费墙未实现**：PRD要求"免费3个名字+付费解锁全部"，但result页面没有任何付费逻辑，所有名字直接展示
6. **同时存在 `pnpm-lock.yaml` 和 `package-lock.json`**：应只保留前者
7. **Supabase + Drizzle 已安装但完全没使用**：如果MVP不用数据库可以先移除，减少bundle体积
8. **缺Error Boundary和统一Loading**：各页面各自处理loading/error状态
9. **`.gitignore`排除了docs**：AGENTS.md提到的 `src/data/` 目录实际不存在
